# ShipVoice 95+ 课程冲分与企业化两阶段路线

## 阶段目标

### Phase 1: 先把课程分数打到 95+

核心原则只有一条: 先把老师最容易扣分的地方补成真实、可复现、可量化的系统，而不是继续堆展示材料。

必须完成的 5 件事:

1. 主链真实化  
   演示主链必须支持真实输入和真实 provider 接入，不能再以 mock ASR / mock LLM / mock TTS 作为默认叙事中心。

2. 基线 vs 改进闭环  
   至少形成一组清晰对照:
   - baseline: 串行级联
   - improved: 安全门控 + RAG + 术语增强 + 可配置真实 provider

3. 低延迟指标真实化  
   区分“模拟链路延迟”和“真实调用耗时”，测量点必须能解释。

4. 语音评测与多轮证据补强  
   保留现有 50 条真实录音与 55 条安全评测，再补一小批多轮上下文样例，让 Part 1 和 Part 2 的衔接更完整。

5. 报告与答辩口径统一  
   所有文档必须统一成一个说法: 这是一个安全感知、证据驱动、可真实接入语音链路的船厂作业助手。

Phase 1 完成标准:

- 演示接口支持音频上传、文本输入、真实 provider 配置和 fallback。
- 评测表里明确区分 baseline / improved / mock / real。
- 答辩时可以直接回答“这套系统现在是否真实可用，哪里是真实，哪里是 fallback”。

### Phase 2: 朝企业级系统推进

在课程 95+ 之后，再做企业化，不反过来。

目标分 6 块:

1. 服务化  
   迁移到 FastAPI + WebSocket，拆分 API、pipeline、provider、evaluation。

2. 数据与审计  
   接 PostgreSQL，保存会话、转写、门控结论、证据、回答、延迟日志。

3. 安全与权限  
   登录、RBAC、操作审计、配置分级、敏感接口隔离。

4. 模型与推理调度  
   ASR / LLM / TTS 独立服务化，带健康检查、超时、回退、缓存。

5. 观测性  
   结构化日志、指标、Tracing、错误告警。

6. 交付工程化  
   Docker、CI/CD、环境配置、部署说明、压测与回归测试。

3. 多轮样例与实验表补强
